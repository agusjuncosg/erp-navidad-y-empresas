¿Qué stack usamos para el backend?

5 usuarios, sin roles, sin migración de datos, sin restricción de costo. Estas son las 3 opciones más sensatas.

A

Supabase + mismo frontend vanilla JS ⭐ Recomendado

Supabase es un Postgres gestionado con API REST y autenticación incluidos. El frontend vanilla JS existente llama a la API de Supabase directamente. Sin servidor propio que mantener.

Pros

  • Cero backend que escribir — Supabase genera la API sola
  • Auth de usuarios incluido (email + password)
  • Dashboard visual para ver/editar datos
  • Frontend: cambios mínimos (reemplazar localStorage por fetch)
  • Deploy del frontend: Netlify o Vercel gratis
  • Supabase Pro: ~$25/mes, muy generoso

Cons

  • Dependencia de un tercero (Supabase)
  • Lógica de negocio compleja va en el frontend o en Edge Functions

Tiempo estimado: 1-2 semanas para tener algo funcional en producción.

B

Railway + Node.js (Express) + PostgreSQL

Un servidor Node Express propio en Railway (PaaS simple), con una base de datos Postgres. Más control que Supabase, menos infraestructura que un VPS.

Pros

  • Control total del backend — podés hacer cualquier lógica
  • Railway es muy fácil de deployar (push to deploy)
  • Stack ampliamente conocido

Cons

  • Hay que escribir el backend (rutas, auth, validaciones)
  • Más código = más tiempo (3-5 semanas)
  • Railway: ~$20-40/mes según uso

Tiempo estimado: 3-5 semanas.

C

VPS (DigitalOcean/Linode) + stack a medida

Un servidor dedicado donde instalás lo que quieras. Máximo control, máxima complejidad operativa.

Pros

  • Control total de la infraestructura
  • Sin límites de ningún servicio
  • Precio fijo predecible (~$12-48/mes)

Cons

  • Mantenimiento del servidor (updates, SSL, seguridad)
  • Más tiempo y conocimiento técnico
  • Overkill para 5 usuarios

Tiempo estimado: 4-8 semanas.
